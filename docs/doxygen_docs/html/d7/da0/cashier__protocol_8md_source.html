1 ## Cashier Protocol
2 
3 This document discusses about how the funds locked and released during the game with the help of notary nodes which are running the cashier protocol.
4 
5 Notary nodes are the nodes which are elected and trusted by the network. The idea of having notary nodes to lock and release the funds is to remove the control over the funds from the dealer.
6 
7 Once if the node is trusted and elected as a notary, it runs the cashier protocol server side. The implementation of cashier portocol is present inside the `privatebet` directory.
8 
9 ### Steps to run a notary node with cashier protocol
10 ```
11 ./bet cashier
12 ```

43 When the game starts since all the trusted notary nodes and the corresponding public keys are preconfigured, each playing nodes identifies the notaries which are active that matches to the criteria set by the dealer. Here assume that if the dealer sets `2 of 4` then each playing node checks if atleast two of the notary nodes are reachable, if so then if the the player has the funds which is greater than or equal to the stack size set by the dealer then the playing node creates an `msig tx` to deposit the funds and shares this `txid` to the dealer.
44 
45 Dealer then verifies whether the `tx` is legitimate or not as mentioned below:
46 ```
47 1. Player sends the joining request, along with it's {public_key}.
48 2. Dealer sends the {stack_size, random_number} to the player.
49 3. Player checks if it has sufficient funds to join the table,
50  3.1 if yes,
51  a. check the status of the notary nodes and deposit stack_size amount to the msig_address
52  b. generates the signature for the random_number
53  c. send {signature(random_number), tx} to the dealer
54  3.2 if no, exit, refill the funds(which is done manually to the address in the chips wallet) and join again if the player wants to play.
55 4. Dealaer verifies the signature(random_number), if yes it verifies the tx as mentioned in step 5 else discard the message.
56 5. Steps to verify the tx.
57  5.1 check if it's unspent.
58  5.2 check if it's sent to the already preconfigured msig_address
59  5.3 check if the amount in tx is equals to stack_size
60 6. if all checks passed in step 5, dealer sends the join_res
61 ```
62 
63 Like this dealer keeps track of the joining players along with lock-in transaction as shown below:
64 ```
65 | player_id | tx |
66 |-----------|------|
67 | 1 | tx_1 |
68 | 2 | tx_2 |
69 | ... | ... |
70 | n | tx_n |
71 ```
72 
73 Once the game is done dealer sends the spending request of these transactions to the winning player.
